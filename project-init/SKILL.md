# 初始化目录结构

## 定位

为小型软件项目创建轻量工作区目录、项目索引和项目级协作规则，作为行业分析师、产品经理、QA、架构师、项目经理、售前工程师和项目负责人的共同文件工作台。

默认面向一个人统筹的 50 万元以下或轻量定制项目。目录要够用，但不把所有角色产物都变成强制流程。

## 默认目录结构

```text
项目名称/
├─ 00_行业分析/
├─ 01_需求/
│  ├─ 原始资料/
│  ├─ 需求探索/
│  └─ 原型设计/
├─ 02_QA评审/
├─ 03_架构设计/
├─ 04_项目计划/
├─ 05_售前方案/
├─ 06_客户确认/
├─ AGENTS.md
└─ README_项目索引.md
```

`00_行业分析` 是可选产物目录。初始化时可以创建，方便需要行业研究的项目直接落报告；但不是所有项目都必须先做行业分析，也不得因此阻塞需求、架构、计划或售前工作。

## 根目录文件职责

- `AGENTS.md`：规定角色职责、产物目录、上下文来源、协作边界和更新规则。只保存相对稳定的工作约定，不写尚未确认或频繁变化的项目背景与目标。
- `README_项目索引.md`：记录项目名称、客户、背景、目标、阶段、最新基线、待确认事项和变更记录，随项目推进持续更新。
- `01_需求/需求探索`：沉淀详细业务背景、业务目标、范围、流程、规则及确认结果。

## 触发语义

用户可以这样调用：

- `初始化新项目：河北省排污权交易系统，客户是 XXX`
- `在 E:\TempProject 下初始化新项目：河北省排污权交易系统`
- `当前文件夹就是项目根目录，初始化轻量项目结构`
- `使用 $project-init 初始化目录结构`

## 路径判断

先判断用户意图：

- 如果用户说“在某路径下初始化新项目：项目名”，在该路径下创建 `项目名`，再创建默认目录结构。
- 如果用户说“当前文件夹就是项目根目录”，直接在当前工作目录创建默认目录结构，不再嵌套项目名。
- 如果用户只给项目名，默认在当前工作目录下创建 `项目名`。
- 如果目标目录已存在，不覆盖已有文件；只补齐缺失目录和缺失索引，并说明保留了已有内容。
- 如果存在多个可能的目标目录且无法判断，不创建，先询问用户确认。

## 目录职责

| 目录 | 内容 | 主要角色 | 是否强制 |
| --- | --- | --- | --- |
| `00_行业分析` | 行业分析报告、行业背景、政策监管、客户画像、行业痛点、数字化机会、能力地图、售前切入点 | 行业分析师 | 可选 |
| `01_需求/原始资料` | 客户邮件、会议纪要、招标文件、截图、参考系统资料 | 用户、产品经理 | 按资料情况 |
| `01_需求/需求探索` | 需求草稿、需求确认包、待确认问题 | 产品经理 | 通常需要 |
| `01_需求/原型设计` | 原型说明、HTML 原型、截图、DEMO 数据 | 产品经理、原型技能 | 按项目需要 |
| `02_QA评审` | 需求、原型、架构、计划、售前方案评审报告 | QA | 按阶段需要 |
| `03_架构设计` | 架构师概要设计、系统边界、页面/API/数据库/流程/技术点清单 | 架构师 | 估算工时时通常需要 |
| `04_项目计划` | 工作量明细、成本测算、排期、风险、交付计划 | 项目经理 | 估算工时时通常需要 |
| `05_售前方案` | 售前提纲、内容稿、PPT、报价页、DEMO 讲解 | 售前工程师 | 按售前需要 |
| `06_客户确认` | 会议纪要、确认清单、客户反馈、最终确认记录 | 项目负责人 | 按沟通情况 |

## 项目级 AGENTS.md 模板

初始化时创建最小版 `AGENTS.md`，不要等待项目背景和目标完成后再创建。内容使用：

```markdown
# 项目协作规则

## 适用范围

本文件适用于整个项目工作区。各角色开展工作前，先读取本文件、`README_项目索引.md` 及与当前任务相关的最新上游产物。

## 项目上下文

- 项目名称、客户、背景、目标、当前阶段和最新基线，以 `README_项目索引.md` 及客户确认记录为准。
- 初始化阶段未知的信息不得自行假设，统一标记为“待确认”。
- 若不同文件存在冲突，以客户确认记录、已确认需求基线及项目索引中登记的最新基线为准；无法判断时列为待确认事项。

## 通用协作规则

- 默认使用中文，输出应结构化、可落地、可评审、可追溯。
- 不删除、不移动、不覆盖已有文件；需要修订时保留原文件或创建明确标识的新版本。
- 产物应注明输入来源、关键假设、范围边界、待确认事项和主要风险。
- 完成或确认新产物后，更新 `README_项目索引.md` 的最新基线、待确认事项和变更记录。
- 不替其他角色擅自作出业务确认、架构承诺、交付承诺或商业承诺。

## 角色职责与产物目录

| 角色 | 主要职责 | 默认产物目录 | 说明 |
| --- | --- | --- | --- |
| 行业分析师 | 梳理行业背景、政策监管、客户画像、行业痛点、数字化机会和售前切入点 | `00_行业分析` | 可选前置输入；不是所有项目强制要求 |
| 产品经理 | 梳理业务问题、目标、范围、流程、功能、规则和原型输入 | `01_需求/需求探索`、`01_需求/原型设计` | 负责把行业假设转成客户确认需求 |
| QA | 独立评审各阶段产物，记录问题、责任角色和复审结果 | `02_QA评审` |  |
| 架构师 | 明确系统边界、架构视图、集成关系、资源假设和概要设计清单 | `03_架构设计` |  |
| 项目经理 | 基于已确认范围和架构概要设计制定工时、成本、排期、资源与风险计划 | `04_项目计划` |  |
| 售前工程师 | 整合项目理解、产品方案、架构、交付范围、计划和报价等客户方案内容 | `05_售前方案` | 可引用行业分析作为方案叙事素材 |
| 项目负责人 | 归档会议纪要、客户反馈、确认清单和最终确认记录 | `06_客户确认` |  |

## 行业分析使用规则

- 行业分析用于售前前置研究，适合陌生行业、新客户、新业务、新赛道或需要客户方案叙事的项目。
- 熟悉行业、需求已经明确或时间紧张的项目，可以跳过行业分析，不得把它作为强制入口。
- 行业分析中的行业共性判断、客户画像和能力地图不得直接视为客户已确认需求；需要产品经理继续确认。
- 售前工程师可以引用行业分析中的背景、痛点、价值主张和案例素材缺口，但不得把未确认假设包装成客户事实。

## 评审与回流

- QA 发现的问题回流到对应责任角色修订，QA 负责复审并记录关闭状态。
- 未确认事项不得包装为已确认范围；对外材料中的假设、可选项和阶段边界必须显式标注。
```

## 执行规则

- 创建目录前先解析目标路径，避免在错误位置创建项目。
- 不删除、不移动、不覆盖用户已有文件。
- 目录创建可以使用系统命令；`README_项目索引.md` 和 `AGENTS.md` 的创建或修改使用文件编辑工具。
- 初始化时同时创建 `README_项目索引.md` 和最小版 `AGENTS.md`；项目背景或目标未知时标记为“待确认”，不得编造。
- 如果 `AGENTS.md` 已存在，保留其中已有的项目专属规则，不覆盖；仅在文件缺失时创建默认模板。对是否补充已有文件存在疑问时，先征求用户确认。
- 初始化完成后返回项目根目录、已创建目录、项目索引和 `AGENTS.md` 路径。
- 不自动启动行业分析、需求探索、QA、架构、项目计划或售前方案；除非用户明确要求继续调用对应角色。

## 质量检查

- 目录结构与轻量模板一致，且包含可选 `00_行业分析`。
- 没有重复嵌套项目名。
- 已有目录和文件被保留。
- `README_项目索引.md` 存在且项目名称、客户名称、项目背景、项目目标、初始化日期和根目录可追踪；未知项已明确标记为“待确认”。
- `README_项目索引.md` 明确行业分析报告是可选产物，不是强制入口。
- `AGENTS.md` 存在，角色职责与默认产物目录和实际目录结构一致。
- `AGENTS.md` 明确行业分析可选、行业假设不得直接当作客户已确认需求。
- 后续角色能够从项目索引找到各自默认产物目录。
