# 轻量成本与工期测算模型

## 1. 适用范围

用于 50 万元以下、3–8 人的小型软件定制项目。输出用于内部立项、资源安排、交付计划和对客报价输入，不替代正式客户报价。

估算采用“规模计数 → 基础人天 → 乐观/基准/悲观三档 → 风险系数 → 配套工作 → 关键路径工期”的轻量方法。信息不足时输出区间和假设，不制造精确数字。

默认采用“基础研发档”，前提是使用团队熟悉的技术栈、成熟后台框架、通用组件和常规部署方式，项目以基础管理、查询、表单、审批和报表功能为主。复杂集成、定制交互、历史数据治理、硬件、私有化适配或新技术验证不提高全部权重，应作为专项工作单列。

## 2. 规模与基础开发人天

### 2.0 规模输入

项目经理使用架构师输出的概要设计 Excel 或清单，不重新推导页面、API 接口概要、数据库概要设计、复杂流程或特殊技术点。输入至少包括：系统模块、页面、API 接口概要、数据库概要表清单、表关系、复杂流程、特殊技术点、专项工作、各明细项复杂度、复杂度原因、下限/基准值/上限、置信度、来源和主要假设。

如果规模清单缺失，先提示调用架构师生成；用户明确要求立即粗估时，可以使用用户提供的数量，但必须标记为 L1、列出缺口并扩大误差区间。

### 2.0.1 标准工时表使用规则

项目经理应维护一份可复用的标准工时表，将架构师给出的简单/中等/复杂规模项转换为基础人天。默认使用本模型内置的“基础研发档”工时表；企业已有标准工时表、行业参数或历史项目基准时，应优先使用企业口径，并记录表名、版本、适用前提和调整原因。

标准工时表只负责把规模明细转换为基础人天，不替代三新风险、联调、测试、部署、客户确认和验收缓冲。

### 2.0.2 工作量明细与汇总

项目经理的工作量估算必须先形成逐项明细，再形成汇总。

工作量明细应逐项列出架构师概要设计中的：

- 每一个页面/视图。
- 每一个 API 接口概要能力。
- 每一个数据库概要表/数据项。
- 每一条复杂流程。
- 每一个特殊技术点。
- 每一个专项工作。

每一行必须包含：

- 编号。
- 工作项。
- 来源类型：页面、API 接口概要、数据库概要设计、复杂流程、特殊技术点、专项工作。
- 所属模块。
- 复杂度。
- 复杂度原因。
- 标准人天。
- 数量/折算系数。
- 基础人天。
- 调整系数。
- 调整后人天。
- 重复计数检查。
- 估算假设。

工作量汇总只能从明细表汇总生成，用于查看各类别简单/中等/复杂数量、基础人天和调整后人天。不得只用聚合数量替代明细估算。

### 2.1 API 接口概要能力

API 接口概要能力按一个可独立开发、联调和验收的后台业务能力或外部协同能力计数，而不是按最终 HTTP 路径计数；增删改查可以按实际复杂度合并或拆分，但必须保持口径一致。

| 复杂度 | 判断参考 | 默认人天/个 |
| --- | --- | ---: |
| 简单 | 单对象增删改查、普通列表/详情、基础权限、规则少 | 0.3 |
| 中等 | 多对象组合、状态校验、常规计算、导入导出或批量操作 | 0.8 |
| 复杂 | 跨系统、复杂状态流、多步骤处理、复杂统计或批处理 | 1.5 |

`E_API = 0.3 × 简单接口数 + 0.8 × 中等接口数 + 1.5 × 复杂接口数`

### 2.2 前端页面

页面按可独立确认和验收的业务视图计数。明显复用同一组件和逻辑的页面可按 0.3–0.7 个折算；PC 与移动端交互不同则分别计算。

| 复杂度 | 判断参考 | 默认人天/页 |
| --- | --- | ---: |
| 简单 | 普通列表、详情、简单表单，使用成熟组件 | 0.5 |
| 中等 | 组合表单、较多校验、权限状态、多区域联动 | 1.0 |
| 复杂 | 看板、复杂图表、可配置分析、多步骤工作台 | 2.0 |

`E_PAGE = 0.5 × 简单页面数 + 1.0 × 中等页面数 + 2.0 × 复杂页面数`

### 2.3 数据库概要设计

数据库概要设计按架构师输出的概要表清单和表关系折算基础人天，重点覆盖数据建模、表设计、表关系梳理、状态字段、审计/历史、导入导出和查询统计的基础工作。

| 复杂度 | 判断参考 | 默认人天/项 |
| --- | --- | ---: |
| 简单 | 单核心对象、少量字典/附件/日志，关系清晰 | 0.3 |
| 中等 | 多核心对象、主从表、多状态、多角色权限、导入导出或审计要求 | 0.8 |
| 复杂 | 多事实来源、复杂汇总统计、历史版本、跨系统同步、迁移清洗、强审计或监管口径 | 1.5 |

`E_DB = 0.3 × 简单数据项数 + 0.8 × 中等数据项数 + 1.5 × 复杂数据项数`

使用约束：

- 数据库概要设计人天用于估算数据层建模和实现复杂度，不等同于最终物理库表设计工作量。
- 如果数据库复杂度已经完全体现在复杂接口、复杂流程或数据迁移专项中，应避免重复计数，并在假设中说明扣减口径。
- 表关系复杂、审计历史要求强、数据迁移或复杂统计较多时，应优先作为中等或复杂项处理。

### 2.4 复杂业务流程

仅对跨角色、改变业务状态、存在重要逆向/异常分支或跨系统协作的流程单列，避免把普通页面操作重复计入。

| 复杂度 | 判断参考 | 默认人天/条 |
| --- | --- | ---: |
| 简单 | 单角色、线性状态变化、异常少 | 1.0 |
| 中等 | 多角色、3–5 个状态、含少量撤回/作废/更正分支 | 2.5 |
| 复杂 | 跨系统/多方协作、多级审批、复杂逆向和指标影响 | 5.0 |

`E_FLOW = 1.0 × 简单流程数 + 2.5 × 中等流程数 + 5.0 × 复杂流程数`

### 2.5 特殊技术点与专项工作

特殊技术点不应只靠接口或页面数量吸收。架构师已标注事务、幂等、MQ/异步、补偿、并发、外部集成、复杂权限、复杂报表、数据迁移等技术点时，项目经理应按影响范围单列附加人天或折算系数。

| 复杂度 | 判断参考 | 默认人天/项 |
| --- | --- | ---: |
| 简单 | 成熟框架内可配置或低风险适配 | 0.5 |
| 中等 | 需要设计、编码、测试和少量联调验证 | 2.0 |
| 复杂 | 影响核心链路、需 POC/联调/补偿/压测或多方确认 | 5.0 |

`E_TECH = 0.5 × 简单技术点数 + 2.0 × 中等技术点数 + 5.0 × 复杂技术点数`

专项工作按实际范围单列，不能被页面、API 接口概要能力和数据库概要设计覆盖的工作包括：

- 历史数据清洗、映射、迁移和业务核对。
- 复杂报表、可编辑图表、BI 或数据模型准备。
- 第三方 SDK、硬件、SSO、消息、支付等集成验证。
- 私有化部署、网络/安全配合、环境初始化。
- 培训、上线支持、操作手册和验收材料。
- POC、样例链路或技术预研。

### 2.6 复杂度使用约束

- 普通增删改查、通用权限、字典、日志、附件和成熟组件复用默认按简单项，不因字段较多自动升级。
- 常规导入导出、批量操作或轻量状态校验通常按中等项；已经由成熟框架直接提供时可按 0.3–0.7 个折算。
- 复杂项必须有明确依据，例如跨系统联动、复杂状态机、多级审批、动态规则、复杂统计或明显定制交互。
- 如果复杂项占接口、页面或流程总数超过 15%，项目经理应提示架构师复核分类或说明复杂性来源。
- 页面、API 接口、数据库、流程和特殊技术点之间已经覆盖的工作不得重复计数；数据迁移、第三方适配、私有化和 POC 仍作为专项工作单列。

### 2.7 不能遗漏的显式工作

下列工作不应硬塞进接口或页面数量，按实际范围单列人天：

- 历史数据清洗、映射、迁移和业务核对。
- 复杂报表、可编辑图表、BI 或数据模型准备。
- 第三方 SDK、硬件、SSO、消息、支付等集成验证。
- 私有化部署、网络/安全配合、环境初始化。
- 培训、上线支持、操作手册和验收材料。
- POC、样例链路或技术预研。

基础开发人天：

`E_BASE = E_API + E_PAGE + E_DB + E_FLOW + E_TECH + E_OTHER`

## 3. 三新系数

分别判断后相乘：`K = K_TEAM × K_TECH × K_BUSINESS`。

| 维度 | 情况 | 系数 |
| --- | --- | ---: |
| 新团队 | 成熟合作、角色稳定 | 1.00 |
| 新团队 | 部分成员首次合作 | 1.08 |
| 新团队 | 全新团队或关键角色不稳定 | 1.15 |
| 新技术 | 团队熟悉且有项目经验 | 1.00 |
| 新技术 | 局部首次使用、有样例可参考 | 1.10 |
| 新技术 | 核心链路首次使用、需要 POC | 1.20 |
| 新技术 | 未验证且强依赖外部条件 | 1.30 |
| 新业务 | 业务熟悉、规则稳定 | 1.00 |
| 新业务 | 部分新场景、关键口径基本清楚 | 1.10 |
| 新业务 | 全新领域或规则/验收尚不稳定 | 1.20 |

`E_DEV = E_BASE × K`

当 `K > 1.40` 时，不给单点承诺；先安排 POC/样例链路，并至少给出基准与保守两个区间。

## 4. 配套工作量

以调整后的开发人天为基础，默认取值如下；已有详细 WBS 时以详细拆分替代比例。

| 工作 | 默认算法 | 常用范围 |
| --- | --- | --- |
| 需求/方案/项目管理 | `E_DEV × 15%` | 12%–20% |
| 测试与缺陷验证 | `E_DEV × 25%` | 20%–30% |
| 接口契约与 Mock | `max(2, 接口数 × 0.15)` | 多方协作可乘 1.5 |
| 联调支持 | `E_DEV × 15%` | 无外部系统 10%，多方/不稳定接口 15%–25% |
| 部署、培训与上线 | `E_DEV × 5%` | 3%–10% |

`E_TOTAL = E_DEV + E_PM_BA + E_QA + E_CONTRACT + E_INTEGRATION + E_RELEASE`

避免重复计数：如果复杂流程人天已经明确包含某项专项联调或迁移，应从配套工作中扣除相同部分，并在假设中说明。

## 5. 估算区间与置信度

| 成熟度 | 可用材料 | 建议误差范围 |
| --- | --- | --- |
| L1 粗估 | 只有功能目录和大致数量 | -20%～+35% |
| L2 方案估算 | 有架构师概要设计清单、需求草稿和原型 | -10%～+25% |
| L3 计划基线 | 需求、原型、正式接口契约和验收口径基本确认 | -5%～+15% |

对外承诺使用悲观值或区间上沿；内部资源计划可使用基准值，并保留风险缓冲。

### 5.1 乐观、基准、悲观三档

项目经理输出估算时，默认给出三档口径：

| 档位 | 用途 | 建议算法 | 适用前提 |
| --- | --- | --- | --- |
| 乐观 | 内部底线、成本下限、快速判断毛利 | `基础或调整后人天 × 0.85～0.90` | 范围冻结、组件可复用、客户确认及时、返工少 |
| 基准 | 资源计划、排期主口径 | `基础或调整后人天 × 1.00` | 需求和原型基本清楚，按常规交付推进 |
| 悲观 | 风险边界、对外承诺、缓冲预留 | `基础或调整后人天 × 1.15～1.25` | 存在确认延迟、口径变更、联调不稳定或验收反复 |

使用规则：

- 用户要求“乐观算”时，明确只代表内部成本或底线测算，不作为对外承诺。
- 用户要求报价或合同边界时，优先使用基准到悲观区间，并说明不含项和客户配合前提。
- 三新系数和三档系数不得机械重复放大；如果三新系数已经覆盖主要风险，悲观档只补充需求变更、联调和验收缓冲。
- 输出表格应同时列出数量、乐观工时、基准工时、悲观工时，除非用户明确只要某一档。

## 6. 内部交付成本

默认使用月度工资法/人员月成本法：先按角色拆分人天，再将月度人员成本折算为人天成本。

核心公式：

`角色人天成本 = 角色月度人员成本 ÷ 月有效工作日`

`C_LABOR = Σ(角色人天 × 角色人天成本)`

`C_DELIVERY = C_LABOR + 云资源/测试环境 + 软件许可 + 差旅 + 第三方费用 + 风险准备`

默认月有效工作日采用 24 天（大小周口径）；用户给出其他企业内部口径时，以用户口径为准。

示例或粗估场景下，角色月度人员成本可使用以下默认值；正式测算以企业实际人员成本为准。

| 角色类别 | 默认月度人员成本 |
| --- | ---: |
| 研发类岗位（架构/技术负责人、后端、前端、实施/运维等） | 10000 元/人月 |
| UI 设计 | 7000 元/人月 |
| 测试 | 7000 元/人月 |

项目经理、产品/需求、售前支持等非研发类角色若未提供企业成本口径，先标记为 `待确认`；用户允许粗估时可按 10000 元/人月临时测算，并在假设中说明。

风险准备通常为直接成本的 5%–10%；三新叠加、范围未确认或外部依赖不稳定时可取 10%–20%。

未提供月度人员成本时，输出各角色人天、月度工资法成本公式和待补参数，不自行填写金额。客户报价、税率、利润和付款条件需由用户确认，并可交给售前工程师整合到方案中。

对比口径：

- 月度工资法用于内部成本价和毛利底线测算。
- 人日单价法用于对外报价、商务测算或合同口径。
- 月度排班法用于判断固定周期内需要多少人，不直接等同于成本价或报价。

## 7. 工期与关键路径

角色有效产能：

`有效产能 = 人数 × 投入比例 × 0.7～0.85`

0.7～0.85 用于扣除沟通、评审、缺陷修复和非项目占用。不要默认每人每天 100% 用于新增开发。

前后端或多方共同开发时采用接口先行：

1. 范围和核心流程确认。
2. 接口契约、字段、错误码、样例数据和 Mock 确认。
3. 前端、后端、客户/第三方、测试用例、环境和数据准备并行推进。
4. 各方达到约定准入标准后进入联合联调。
5. 联调通过后进行系统测试和回归。
6. 系统测试通过后进入客户 UAT、缺陷关闭和上线准备。
7. 完成上线、观察和验收。

简化关键路径公式：

`D_TOTAL = D_SCOPE + D_CONTRACT + max(D_BACKEND, D_FRONTEND, D_PARTNER, D_ENV_DATA) + D_INTEGRATION + D_TEST + D_UAT + D_RELEASE + D_BUFFER`

默认阶段区间：

| 阶段 | 常用日历工作日 | 串并行规则 |
| --- | ---: | --- |
| 范围/核心流程确认 | 3–7 | 关键前置，原则上串行 |
| 接口契约与 Mock | 2–5 | 核心接口确认后才能稳定并行开发 |
| 各方开发 | 按角色人天/有效产能计算 | 前端、后端、第三方并行 |
| 联调 | 5–10 | 等各方达到准入标准；可按模块滚动开始 |
| 系统测试与回归 | 5–10 | 完整联调后为主，可与缺陷修复并行 |
| UAT 与业务确认 | 5–10 | 依赖系统测试通过和业务人员到位 |
| 上线准备与发布 | 2–5 | 依赖关键缺陷关闭和上线条件满足 |
| 关键路径缓冲 | 关键路径的 10%–15% | 外部依赖不稳定时取上沿 |

必须串行的通常包括：范围基线 → 核心接口契约 → 完整联调 → 系统测试准出 → UAT准出 → 上线。可并行的通常包括：前后端开发、第三方开发、测试用例准备、环境准备、基础数据准备和文档编写。

## 8. 输出要求

每次估算至少给出：

1. 输入规模及复杂度口径。
2. 架构师概要设计清单的版本、区间、置信度和假设摘要。
3. 工作量明细：逐项列出页面、API 接口概要、数据库项、复杂流程、特殊技术点和专项工作，并逐项计算基础人天和调整后人天。
4. 工作量汇总：从明细表按来源类型和复杂度汇总数量、基础人天和调整后人天。
5. 基础人天、三新系数、乐观/基准/悲观三档、配套人天和总人天区间。
6. 按角色人天及 3–8 人团队配置建议。
7. 按月度工资法计算的内部成本结果或计算公式。
8. 接口先行的串并行计划和关键路径。
9. 工期基准值、保守值及置信度。
10. 客户/第三方依赖、主要假设和不含项。

推荐 Excel sheet 结构：

| Sheet | 名称 | 内容 |
| --- | --- | --- |

## 9. 快速算例

假设工作量明细逐项列出后，汇总结果为：12 个简单接口、8 个中等接口、4 个复杂接口；8 个简单页面、5 个中等页面、2 个复杂页面；6 个简单数据项、4 个中等数据项、1 个复杂数据项；1 条中等流程、2 条复杂流程；2 个中等特殊技术点：

- `E_API = 12×0.3 + 8×0.8 + 4×1.5 = 16 人天`
- `E_PAGE = 8×0.5 + 5×1 + 2×2 = 13 人天`
- `E_DB = 6×0.3 + 4×0.8 + 1×1.5 = 6.5 人天`
- `E_FLOW = 1×2.5 + 2×5 = 12.5 人天`
- `E_TECH = 2×2 = 4 人天`
- `E_BASE = 52 人天`，未计其他专项工作。

若部分新团队 `1.08`、局部新技术 `1.10`、部分新业务 `1.10`，则 `K≈1.31`，`E_DEV≈68 人天`。再按项目管理/需求 15%、测试 25%、接口契约约 4 人天、联调 15%、部署上线 5% 计算，总量约 113 人天。该结果属于 L1/L2 估算，应结合详细清单给出区间，而不是直接承诺 113 人天。
