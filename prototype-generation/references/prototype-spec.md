# 原型说明文档规范

## 文档结构

完整原型说明文档包含以下部分：

1. 原型目标与范围
2. 用户角色与使用场景
3. 页面/模块清单
4. 页面流与业务流程
5. 页面详细说明
6. 角色权限与数据范围
7. 样例数据说明
8. 待确认问题清单
9. 原型交付说明

## 页面编号

使用统一编号方便交叉引用：

| 类型 | 编号格式 | 示例 |
|---|---|---|
| 页面 | PAGE-{域缩写}-{三位序号} | PAGE-CON-001 |
| 流程 | FLOW-{域缩写}-{三位序号} | FLOW-CON-001 |
| 角色 | ROLE-{角色缩写} | ROLE-SALES |
| 状态 | STATE-{域缩写}-{三位序号} | STATE-CON-001 |
| 待确认问题 | Q-{三位序号} | Q-001 |

## 页面说明结构

每个页面使用以下结构：

```markdown
### PAGE-CON-001 合同列表

**页面目的**：帮助业务人员查询和跟踪合同处理状态。

**主要用户**：ROLE-SALES 销售人员、ROLE-MANAGER 部门经理。

**入口来源**：工作台、合同管理菜单。

**展示信息**：
- 合同编号
- 客户名称
- 合同金额
- 合同状态
- 责任人
- 最近更新时间

**主要操作**：
- 新增合同
- 查看详情
- 提交审批
- 筛选合同

**状态与反馈**：
- 空状态：暂无合同数据时提示创建合同。
- 权限受限：无新增权限时隐藏新增操作。
- 错误状态：查询失败时保留筛选条件并提示重试。

**跳转关系**：
- 新增合同 → PAGE-CON-002 合同创建
- 查看详情 → PAGE-CON-003 合同详情

**样例数据**：使用虚构客户、虚构合同编号和脱敏金额区间。
```

## HTML 原型验收标准

可点击 HTML 原型至少满足：

- 首屏就是可使用的业务界面，而不是介绍型落地页。
- 包含清晰导航，能在核心页面之间切换。
- 每个核心流程至少有一条可点击路径。
- 列表、详情、表单、审批、统计等页面使用真实感样例数据。
- 操作会产生可见反馈，例如状态变更、提示、弹窗或页面切换。
- 关键空状态、错误状态、权限受限状态至少覆盖与业务最相关的部分。
- 页面在桌面和移动宽度下不发生明显文字溢出或控件重叠。
- 不使用真实敏感客户、个人、合同、金额数据，除非用户明确提供并允许。

## 管理后台原型额外要求

当原型属于管理后台、运营后台、内部管理系统、CRM、ERP、OA、数据管理平台、审批后台或权限管理后台时，必须额外满足：

- 首屏应是工作台、列表页或业务处理页，不做营销式首页。
- 采用稳定的信息架构：左侧主导航、顶部全局操作区、主内容工作区，移动端可折叠导航。
- 页面应以表格、表单、详情、审批、配置、日志、统计为核心组件。
- 列表页必须说明筛选区、表格列、排序、分页、行内操作、批量操作、空状态和权限受限状态。
- 表单页必须说明字段分组、必填项、校验规则、保存草稿、提交、取消、错误反馈。
- 详情页必须说明基础信息、状态、关联记录、操作区、操作日志或流转记录。
- 审批页必须说明审批意见、通过、驳回、转交/加签（如适用）、驳回后去向。
- 权限相关页面必须区分菜单权限、操作权限、数据范围，不可只写“管理员可见”。
- 危险操作必须有二次确认或明确的撤销/恢复路径说明。
- 交互反馈必须覆盖成功、失败、加载、空数据、无权限、网络异常或系统异常。
- 视觉风格应克制、清晰、便于长时间使用，避免大面积装饰性视觉、营销文案和过度卡片化。
