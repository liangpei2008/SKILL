# 架构师概要设计清单规则

## 1. 目标与边界

从需求文件和原型形成售前规模输入，供项目经理测算人天、内部成本、团队和工期。允许因需求成熟度产生合理偏差，必须给出来源、假设、下限、基准值、上限和置信度。

不定义正式接口 URL、HTTP 方法、字段全集、错误码、认证细节、样例、Mock、页面交互规格、物理数据表或技术实现。

架构师概要设计产物默认输出为 Excel 工作簿，除非用户明确要求其他格式。Excel 应拆分多个 sheet，保证页面、API、数据库、流程、特殊技术点和风险可以独立筛选、汇总和追溯。

## 1.1 小项目默认复杂度口径

面向 50 万元以下、基础功能研发为主的小项目时，优先使用以下轻量判断：

| 复杂度 | 默认判断 | 典型情况 |
| --- | --- | --- |
| 简单 | 单对象、低规则、低状态、无外部依赖 | 单表或单业务对象增删改查、普通列表、详情、基础维护 |
| 中等 | 多对象、有状态、有权限、有校验、有导入导出或简单计算 | 多表组合维护、主从表、状态动作、权限校验、批量导入导出 |
| 复杂 | 跨系统、多角色强协同、多步骤事务、复杂统计、批处理、补偿、结算或监管影响 | 第三方系统集成、复杂看板、绩效/财务/监管口径、多级审批、跨系统失败处理 |

使用规则：

- 单表或单业务对象增删改查，通常判为简单。
- 多表或多业务对象组合操作，通常判为中等。
- 跨系统、复杂统计、多步骤事务、复杂状态流、批处理、失败补偿、结算或监管影响，通常判为复杂。
- 单表功能如果包含复杂状态流、权限隔离、历史版本、审计追溯或复杂校验，可以升为中等。
- 多表功能不自动判为复杂；只有出现跨系统、多方协同、复杂统计、补偿、结算或监管影响时才升为复杂。
- 判定复杂度时必须记录触发因素，避免只写“复杂”但没有依据。

## 2. 页面规模

按可独立确认和验收的业务视图计数，区分 PC、移动端和大屏/看板。

| 复杂度 | 判断参考 |
| --- | --- |
| 简单 | 普通列表、详情、简单表单、静态说明 |
| 中等 | 组合表单、复杂校验、权限状态、多区域联动 |
| 复杂 | 看板、复杂图表、可配置分析、多步骤工作台 |

计数规则：

- 同一路由下用途明显不同的列表、详情、维护视图可分别计数。
- 同一组件换角色或换入口但功能相同，按复用比例折算，不重复按完整页面计算。
- PC 与移动端交互明显不同则分别计算。
- 没有原型时，根据角色、功能和流程推导候选页面，置信度最高只能标为 L1。

前端估工口径：

- 页面数量按“可独立确认和验收的业务视图”统计，但给项目经理估工时不得直接等同为完全独立前端页面数量。
- 默认采用“核心页面骨架 + 可复用前端组件 + 业务字段适配”评估前端工作量。
- 通用列表、详情、表单、筛选、指标卡、图表、排名表、导入、导出、预警、状态标签、操作日志、确认弹窗、抽屉等应作为组件复用能力统一估算。
- 列表页中的新增/编辑弹窗、删除确认、启用/停用确认、筛选抽屉、普通导入导出弹窗，不单独按完整页面计数；并入对应页面或组件适配。
- 只有具备独立路由、独立验收口径、明显不同交互或多步骤工作台特征的视图，才按独立页面骨架估算。
- 看板类页面应识别可复用的筛选器、指标卡、图表、排名表和钻取组件；多个看板复用同一数据展示组件时，只把首个基准看板按复杂页处理，其余按中等或复用折算。
- 输出规模汇总时，建议同时给出“业务视图数量”“独立页面骨架数量”“组件复用说明”和“复杂页面保留项”，避免页面数量被高估。

## 3. API 接口概要规模

按一个可独立开发、联调和验收的后台业务能力或外部协同能力计数，不按最终 HTTP 路径计数。

能力类型包括：查询、维护、状态动作、计算/统计、导入、导出、消息/外部协同。

推导规则：

- 普通业务对象通常考虑列表查询、详情查询、新增/修改，以及必要的停用/作废能力。
- 每个规则和结果不同的状态动作独立计数，例如提交、暂停、恢复、作废、复筛和关闭。
- 导入、导出、批量校验、复杂统计、看板聚合、消息播报和外部协同独立计数。
- 多个页面复用同一后台能力时只计算一次；同一页面需要多个独立能力时分别计算。
- 静态展示、纯前端计算或完全复用既有能力的页面不新增 API 接口概要能力。
- 无法判断复用时，同时给出去重后下限和按独立能力计算的上限。

输出规则：

- API 接口概要清单不定义 URL、HTTP 方法、字段全集、错误码、认证细节、样例或 Mock。
- 每个接口能力必须标注复杂度和复杂度原因，避免只写“复杂”但没有依据。
- 每个接口能力的复杂度和复杂度原因均为必填项；无法判断时复杂度标为“待确认”，复杂度原因写明缺口来源。
- 涉及事务一致性、幂等控制、MQ/异步处理、失败重试、补偿机制、并发控制、批量处理、跨系统调用、状态机流转、审计日志或高频调用时，必须在对应接口能力或业务动作中标出。

复杂度参考：

| 复杂度 | 判断参考 |
| --- | --- |
| 简单 | 单对象增删改查、普通列表/详情、规则少 |
| 中等 | 多对象组合、权限/状态校验、计算规则、导入导出 |
| 复杂 | 跨系统、复杂状态流、多步骤处理、复杂统计或批处理 |

## 4. 数据库概要设计规模

数据库概要设计用于辅助评估数据建模、表设计、表关系、查询统计、导入导出、数据迁移、权限隔离、审计日志等工作量，不作为开发阶段物理库表设计。

必须同时输出概要表清单和表之间的关系，不得只列业务实体或孤立表。

概要表清单至少包含：

- 核心业务实体。
- 建议表。
- 核心字段类别。
- 关键状态字段。
- 数据来源/维护方。
- 预计数据量级。
- 历史数据处理。
- 审计留痕。
- 待确认事项。

表关系至少说明：

- 一对一、一对多、多对多、主从、引用、汇总、日志/流水、附件等关系。
- 关系依据。
- 关系中的关键业务约束。
- 待确认项。

复杂度参考：

| 复杂度 | 判断参考 |
| --- | --- |
| 简单 | 单核心对象、少量字典/附件/日志，关系清晰 |
| 中等 | 多核心对象、主从表、多状态、多角色权限、导入导出或审计要求 |
| 复杂 | 多事实来源、复杂汇总统计、历史版本、跨系统同步、迁移清洗、强审计或监管口径 |

## 5. 复杂业务流程

只统计跨角色、改变业务状态、存在重要逆向/异常分支或跨系统协作的流程，避免把普通页面操作重复计入。

| 复杂度 | 判断参考 |
| --- | --- |
| 简单 | 单角色、线性状态变化、异常少 |
| 中等 | 多角色、3–5个状态、含撤回/作废/更正等分支 |
| 复杂 | 跨系统/多方协作、多级审批、复杂逆向和指标影响 |

## 6. 特殊技术点与专项工作规模

特殊技术点应单独汇总，不能只散落在接口说明中。

至少说明：

- 技术点名称。
- 涉及模块/接口。
- 工作量影响。
- 主要风险。
- 待确认事项。

常见特殊技术点包括：

- 事务一致性。
- 幂等控制。
- MQ/异步处理。
- 失败重试。
- 补偿机制。
- 并发控制。
- 批量处理。
- 跨系统调用。
- 状态机流转。
- 审计日志。
- 高频调用。
- 外部集成。
- 复杂权限。
- 复杂报表。
- 数据迁移。

专项工作规模单列不能被页面、API 接口概要能力和数据库概要设计覆盖的工作：

- 历史数据清洗、映射、迁移和业务核对。
- 复杂报表、可编辑图表、BI或数据模型准备。
- 第三方SDK、硬件、SSO、消息等集成验证。
- 私有化部署、网络安全配合和环境初始化。
- 培训、上线支持、操作手册和验收材料。
- POC、样例链路或技术预研。

专项工作只描述规模和复杂度，不在架构师阶段换算人天。

## 7. 置信度与区间

| 等级 | 材料成熟度 | 建议规模偏差 |
| --- | --- | --- |
| L1 粗估 | 功能目录或不完整需求，无原型 | -20%～+35% |
| L2 售前估算 | 需求草稿、核心流程和原型基本具备 | -10%～+25% |
| L3 计划输入 | 需求、原型、边界和验收口径基本确认 | -5%～+15% |

售前通常使用 L1 或 L2。基准值用于内部测算，上限用于风险和报价边界；不得把上限或基准值表述为最终开发承诺。
